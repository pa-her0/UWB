# UWB 数据库接入 QT 软件需求说明

## 1. 目标

将当前项目已有的 `UWB CSV -> InfluxDB` 入库能力和 `按时间范围查询轨迹` 能力，真正接入到 QT 软件中，形成一个可独立使用的“UWB 数据可视化页面”。

最终目标不是只做一次性脚本展示，而是让 QT 软件具备以下能力：

1. 自动连接并读取 InfluxDB 中的 UWB 数据。
2. 在 UI 中提供独立页面完成时间范围查询、标签筛选、轨迹展示和基础数据查看。
3. 支持将查询结果可视化，用于分析某一时间段内的运动轨迹和状态变化。
4. 保证后续可扩展到更多字段展示、更多图表和更多设备来源。

## 2. 当前工程现状

当前仓库已经具备以下基础能力：

1. 可将 `log.csv` 解析后写入 InfluxDB。
2. 当前核心 measurement 为 `uwb_tag_telemetry`。
3. 已经有按 `tag_id + 时间范围` 查询 `position_x_m / position_y_m` 并输出 SVG 轨迹图的脚本能力。
4. 数据模型已经初步稳定，适合作为 QT 侧读取接口的后端数据源。

因此，QT 接入阶段的重点不再是“怎么存”，而是“怎么稳定读、怎么展示、怎么交互、怎么工程化管理”。

## 3. 建设范围

本次 QT 接入建议拆成 4 个子模块：

1. 数据源配置模块
2. 数据查询服务模块
3. 可视化页面模块
4. 运行状态与异常处理模块

## 4. 总体业务场景

用户在 QT 软件中进入“UWB 数据可视化”页面后，可以完成以下流程：

1. 系统自动读取数据库连接配置并尝试连接 InfluxDB。
2. 页面展示数据库连接状态、可用 Tag 列表、可查询时间范围。
3. 用户选择某个 `tag_id`，选择开始时间和结束时间。
4. 用户点击“查询”后，系统从 InfluxDB 拉取该时间段的数据。
5. 页面展示二维轨迹图，并可同步显示关键数值或明细表格。
6. 用户可缩放、平移、查看某一点的时间和值，必要时导出图片或导出数据。

## 5. 功能需求

### 5.1 数据库连接管理

QT 软件需要提供数据库接入配置能力，至少包含以下参数：

1. InfluxDB URL
2. Org
3. Bucket
4. Token
5. Measurement 名称
6. 默认查询时间范围

功能要求：

1. 软件启动时自动加载本地配置。
2. 支持“测试连接”按钮。
3. 支持显示当前连接状态：未连接、连接中、连接成功、连接失败。
4. 连接失败时，页面要给出明确原因，例如 URL 错误、Token 无效、Bucket 不存在、查询超时。
5. 配置文件应独立保存，便于后期部署和切换环境。

### 5.2 数据自动读入

这里的“自动读入”建议明确为两层能力：

1. 自动连接数据库并查询已有数据
2. 自动刷新页面中的最新数据或最新时间范围

功能要求：

1. 页面打开时自动完成一次默认查询。
2. 支持手动刷新。
3. 支持可配置的自动刷新开关，例如每 1 秒、5 秒、10 秒刷新一次。
4. 自动刷新只针对当前筛选条件重新查询，避免全量拉取历史数据。
5. 若数据库暂时不可用，系统应保留上一次成功结果，并提示当前为缓存/旧结果。

### 5.3 查询条件管理

页面至少要支持以下查询条件：

1. `tag_id` 选择
2. 开始时间
3. 结束时间
4. 查询字段类型

建议字段类型分为：

1. 轨迹类：`position_x_m`, `position_y_m`, `position_z_m`
2. 姿态类：`roll`, `pitch`, `yaw`
3. 惯导类：`accel_*`, `gyro_*`, `mag_*`
4. 测距类：`range0_m` 到 `range7_m`
5. 信号类：`rx_pwr_dbm`

功能要求：

1. 时间选择必须支持日期时间级别，不只到日期。
2. 开始时间不得晚于结束时间。
3. 查询前要做参数校验。
4. 如果未选择 `tag_id`，可支持“全部 Tag”模式，但默认建议单 Tag 查询。
5. 对超大时间跨度查询要给出限制或警告，避免前端卡死。

### 5.4 Tag 列表与时间范围探测

为了提升可用性，系统应自动从数据库中探测基础元信息：

1. 当前有哪些 `tag_id`
2. 每个 `tag_id` 的最早时间和最晚时间
3. 当前数据库内是否存在可查询数据

功能要求：

1. 页面初始化时自动加载 Tag 列表。
2. 当用户切换 Tag 时，自动更新该 Tag 的可查询时间范围提示。
3. 若 Tag 无数据，要在 UI 中明确显示。

### 5.5 轨迹可视化

这是页面的核心能力。

二维轨迹页至少要支持：

1. 按时间范围绘制 `X-Y` 轨迹
2. 区分起点和终点
3. 显示基站 Anchor 位置
4. 鼠标悬停查看轨迹点详情
5. 显示时间轴上的采样点信息

建议展示元素：

1. 轨迹折线
2. 起点标记
3. 终点标记
4. Anchor 标记与名称
5. 坐标轴和网格
6. 图例

可选增强功能：

1. 按时间渐变着色
2. 按速度或信号强度着色
3. 支持缩放、拖拽和平移
4. 支持框选局部区域
5. 支持重置视图

### 5.6 明细数据表

除了图，还建议保留一个表格区域用于辅助分析。

表格至少支持显示：

1. 时间戳
2. `tag_id`
3. `position_x_m`
4. `position_y_m`
5. `position_z_m`
6. `roll/pitch/yaw`
7. `rx_pwr_dbm`

功能要求：

1. 图表和表格联动。
2. 点击图上的点时，表格定位到对应记录。
3. 点击表格行时，图上高亮对应点。
4. 支持分页或虚拟滚动，避免大数据量卡顿。

### 5.7 多图视图扩展

如果你后续希望这个页面长期可用，建议在需求中直接预留扩展位。

推荐页面结构不是只做一个轨迹图，而是支持以下子视图切换：

1. 轨迹视图
2. 姿态曲线视图
3. 测距曲线视图
4. 信号强度视图
5. 原始数据表视图

这样后续不用重做页面框架，只扩展查询字段和图表组件即可。

### 5.8 导出能力

建议作为正式需求纳入：

1. 导出当前图表为 PNG/SVG
2. 导出当前查询结果为 CSV
3. 导出当前查询条件和截图用于分析留档

## 6. UI 页面需求

建议新增一个独立页面，名称可为：

1. `UWB 数据分析`
2. `UWB 轨迹可视化`
3. `时序数据查询`

推荐页面布局如下：

### 6.1 页面顶部

1. 数据源状态显示
2. 当前数据库名称或连接标识
3. 刷新按钮
4. 自动刷新开关

### 6.2 左侧筛选区

1. `tag_id` 下拉框
2. 开始时间选择器
3. 结束时间选择器
4. 查询类型选择器
5. Anchor 显示开关
6. 查询按钮
7. 导出按钮

### 6.3 中间主展示区

1. 默认显示轨迹图
2. 支持切换为折线图或其他视图
3. 支持无数据占位提示
4. 支持加载中动画

### 6.4 底部或右侧明细区

1. 数据表格
2. 当前点详情
3. 查询统计信息

统计信息建议包括：

1. 当前时间范围内的数据点数量
2. 轨迹总长度
3. 平均采样间隔
4. 起止时间
5. 最大/最小坐标范围

## 7. 工程化需求

### 7.1 模块分层

QT 工程内建议明确分层，不要把数据库查询、数据转换、图表绘制都写在界面类里。

推荐分层：

1. `UI 层`
   负责页面控件、用户交互、状态展示。
2. `ViewModel / Controller 层`
   负责接收 UI 参数、触发查询、整理展示数据。
3. `DataService 层`
   负责构造 Flux 查询、访问 InfluxDB、返回结构化结果。
4. `Model 层`
   负责定义轨迹点、查询条件、查询结果、统计信息等数据结构。
5. `Config 层`
   负责数据库配置、默认参数、Anchor 配置加载。

### 7.2 配置管理

建议以下配置独立管理：

1. 数据库连接配置
2. 默认查询配置
3. Anchor 坐标配置
4. 自动刷新配置
5. 导出路径配置

配置要求：

1. 支持本地文件保存，例如 JSON 或 INI。
2. 软件启动时自动加载。
3. 配置缺失时提供默认值。
4. 配置错误时给出可定位的错误信息。

### 7.3 接口设计要求

建议 QT 侧至少抽象出以下接口：

1. `connectDatabase()`
2. `testConnection()`
3. `loadTagList()`
4. `queryTrajectory(tagId, startTime, endTime)`
5. `queryTelemetry(tagId, startTime, endTime, fields)`
6. `loadAnchors()`
7. `exportCurrentView()`

返回值不要直接面向 UI 控件，而应面向结构化数据对象。

### 7.4 性能要求

1. 普通查询响应时间目标不超过 2 秒。
2. 大时间范围查询时应支持分页、限流或降采样。
3. 图表渲染要避免一次性绘制过多点导致界面卡顿。
4. 自动刷新时不能阻塞主线程。
5. 数据查询和图表绘制应放在异步线程或后台任务中处理。

### 7.5 日志与可维护性

建议增加运行日志，至少记录：

1. 数据库连接结果
2. 查询参数
3. 查询耗时
4. 返回数据量
5. 异常信息

这对后期排查“为什么页面没数据”非常关键。

## 8. 数据接口需求

结合当前仓库数据模型，QT 侧至少需要面向以下逻辑对象：

### 8.1 查询条件对象

包含：

1. `tag_id`
2. `start_time`
3. `end_time`
4. `fields`
5. `limit`
6. `downsample_interval`

### 8.2 轨迹点对象

包含：

1. `time`
2. `x`
3. `y`
4. `z`
5. `tag_id`

### 8.3 Anchor 对象

包含：

1. `name`
2. `x`
3. `y`
4. 可选 `z`

### 8.4 查询结果对象

包含：

1. 轨迹点列表
2. 曲线数据列表
3. 原始表格数据
4. 统计摘要
5. 查询是否成功
6. 错误消息

## 9. 交互与状态要求

页面至少要覆盖以下状态：

1. 初始未加载
2. 正在连接数据库
3. 数据加载中
4. 查询成功且有数据
5. 查询成功但无数据
6. 查询失败
7. 自动刷新中

每个状态都要有明确 UI 表现，不能只有空白页。

## 10. 异常场景需求

以下异常要纳入需求，不然后面落地时会很痛苦：

1. 数据库未启动
2. URL 不可访问
3. Token 错误
4. Bucket / Org 配置错误
5. 时间范围内无数据
6. 查询结果字段缺失，例如只有 `x` 没有 `y`
7. Anchor 配置文件缺失或格式错误
8. 查询数据量过大导致前端卡顿
9. 自动刷新过程中连接中断

对应处理原则：

1. 页面提示要明确且可读。
2. 不允许程序崩溃。
3. 异常日志要可追踪。
4. 用户应能重新发起连接或刷新。

## 11. 非功能需求

### 11.1 易用性

1. 默认配置尽量可直接运行。
2. 常用参数不要让用户每次重复输入。
3. 页面结构清晰，避免把数据库配置和分析操作混在一起。

### 11.2 可扩展性

1. 后续可扩展更多 measurement 或更多设备类型。
2. 后续可扩展实时流式显示，而不推翻当前架构。
3. 后续可扩展三维轨迹显示。

### 11.3 可测试性

1. 查询服务层应能脱离 UI 做单元测试。
2. 配置加载应能单独测试。
3. Flux 查询构造逻辑应能单独验证。

## 12. 建议的版本范围

为了更稳，建议按两个阶段推进。

### 12.1 V1 必做

1. 连接 InfluxDB
2. 加载 Tag 列表
3. 选择时间范围
4. 查询单个 Tag 的二维轨迹
5. 显示 Anchor
6. 显示明细表
7. 支持导出图片或 CSV

### 12.2 V2 增强

1. 自动刷新
2. 多图视图切换
3. 姿态/测距/信号曲线
4. 查询统计分析
5. 降采样和大数据量优化
6. 多 Tag 对比显示

## 13. 对你当前设计最有价值的落地建议

如果你现在要开始画 QT 页面和设计类结构，优先定这 5 件事：

1. 先定义一个独立页面 `UwbVisualizationPage`
2. 再定义一个独立服务类 `InfluxQueryService`
3. 再定义统一的数据对象 `TrajectoryPoint / QueryCondition / QueryResult`
4. 图表区和表格区必须做联动
5. 查询逻辑必须异步化，不能写死在 UI 线程

## 14. 一句话总结

这次 QT 接入的本质，不是“把数据库数据显示出来”这么简单，而是要做成一个具备 `连接管理、条件查询、轨迹可视化、数据联动、异常处理、可扩展演进` 的完整分析页面。

如果按这个需求去设计，你后面无论是先做 V1 演示版，还是继续扩展成正式分析模块，结构都不会乱。
